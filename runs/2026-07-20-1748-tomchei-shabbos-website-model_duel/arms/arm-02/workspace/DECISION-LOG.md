## P4 — order builder, address book, customer account

- **DECISION-P4-1 (draft ≠ Order):** builder drafts are a new `OrderDraft` row holding the cart as validated JSON. Mid-build lines have no recipient, which would break Order/OrderLine's finalize invariants; P5 checkout converts ACTIVE drafts into real Orders and marks them COMPLETED (the only guest-clear path).
- **DECISION-P4-2 (customer auth):** mirrors P1 dev-mode staff auth (scrypt + HMAC-hashed DB session) in a separate table + cookie (`tomchei_customer`) so customer and staff identity can never cross. Clerk path unchanged behind AUTH_MODE=clerk (no keys in harness). Seed credential: sample.customer@example.com / customer-demo-1234.
- **DECISION-P4-3 (no external address/geocode APIs):** no API keys exist in this environment. Autocomplete = customer's saved book + a local delivery-area street index; geocode = local ZIP-centroid provider through the GeocodeCache contract (R-162). Each is one function to swap for a real provider.
- **DECISION-P4-4 (on-order semantics):** the "on-order" pick means the address typed on this order (usually the buyer), stored once per draft as `cart.onOrderRecipient` — it is not saved to the address book. The assignment dialog states this and warns before an edit re-addresses already-assigned on-order lines.
- **DECISION-P4-5 (issues don't block autosave):** `priceCart` reports stock/option/restriction problems per line but saving never fails on them — autosave must not lose work; checkout (P5) is the gate that refuses a cart with issues.

## P4 fix pass

- **DECISION-P4-6 (registration matches by email only):** `findOrLinkCustomer` no longer matches or links by phone. Possessing a phone number proves nothing, so a phoneless staff-created customer record must not be claimable by whoever types that number at registration (B1). Phone-based dedupe of staff-entered orders becomes a staff-side concern; a phone number already on another record is stored raw-only (unique `phoneNormalized` stays consistent).
- **DECISION-P4-7 (register is anti-enumeration, like login):** registration always answers `200 {ok:true}`. Fresh/passwordless email → password set + session; already-registered email → no state change (unless the supplied password happens to be correct, which is just a sign-in). The old `409 "already exists"` confirmed which emails hold accounts. Residual: session-cookie presence still differs — full indistinguishability needs an email-verification flow, out of scope for dev auth.
- **DECISION-P4-8 (X-Forwarded-For is untrusted by default):** `clientIp` ignores XFF unless `TRUST_PROXY=true` (deployed behind exactly one reverse proxy, in which case the last hop — the one the proxy appended — is used). Direct-served dev shares one `"direct"` rate-limit bucket; limits were sized to tolerate that.
## P5 — checkout, delivery fees, Stripe, order lifecycle

- **DECISION-P5-1 (no Stripe keys → mock gateway, real path):** the harness has no Stripe account. The gateway is real REST-over-fetch when STRIPE_SECRET_KEY is set; without it, a mock hosted-checkout page signs and posts checkout.session.completed through the REAL webhook route, so signature verification, idempotency, amount safety, finalize, and refunds run identical code in both modes. No stripe npm package (resolution 8b + dependency ladder).
- **DECISION-P5-2 (order created at pay-time, stock reserved at webhook):** checkout converts the draft into a DRAFT Order with full price snapshots but reserves NO stock; reservation happens inside finalizeOrder when the payment webhook lands. An abandoned Stripe session never holds inventory. A retry for the same builder draft discards and replaces its predecessor order (one draft = one payable order).
- **DECISION-P5-3 (refunds are negative payment rows):** refunds land as negative-amount POSTED Payment rows keyed by the unique stripeRefundId — recalcPaymentStatus needs no changes and refund sync is idempotent by construction.
- **DECISION-P5-4 (guest cookie cleared on success page):** webhooks have no browser cookies, so the webhook completes the OrderDraft row and the success page drops the guest cookie after verifying the cookie's own draft is COMPLETED.
- **DECISION-P5-5 (fee semantics):** bulk delivery bills per distinct destination ADDRESS (two recipients at one address share a fee); per-package bills per RECIPIENT with the hard zip block and a required Purim-week day. Shipping uses the first configured flat rate as the placeholder until P8's Shippo margin engine (R-032).
- **DECISION-P5-6 (charged-amount safety on both sides):** the webhook requires amount_total to equal both the session's amount and the order's current total, and the order to still be a live DRAFT with an open session; anything else is auto-refunded in full and the order discarded, with BOTH sides (charge in, refund out) recorded so the books match Stripe.

## P5 fix pass

- **DECISION-P5-7 (fail-closed Stripe config):** startup refuses two misconfigurations that would silently corrupt the money path: STRIPE_SECRET_KEY set while STRIPE_WEBHOOK_SECRET is still the repo-committed dev default (webhook forgeable by anyone), and NODE_ENV=production without STRIPE_SECRET_KEY (mock gateway "accepting" payments no one made). `getPaymentGateway` re-checks the production case as defense in depth. The build phase (`NEXT_PHASE=phase-production-build`) is exempt because no runtime env exists yet.
- **DECISION-P5-8 (webhook ledger is pending→processed):** the event-id row is inserted as `pending` and flipped to `processed` only after the handlers commit. A crash mid-work means Stripe's retry finds a pending row and reprocesses instead of being swallowed as a replay; handlers are retry-safe (charge keyed by payment intent, refunds keyed by refund id, finalize skipped when already done). Residual: two *concurrent* deliveries of one pending event could double-process in a narrow window — Stripe retries are sequential, accepted.
- **DECISION-P5-9 (refund failure is a visible state):** when an auto-refund's gateway call fails, the session is marked `refund_failed` (never `auto_refunded`) and the order is left untouched for ops to reconcile; the charge row is still booked so the ledger matches Stripe. Refund sync listens for `charge.refunded` / `charge.refund.updated` (the events Stripe actually emits), using the embedded refunds list when present or the cumulative `amount_refunded` delta under a deterministic key otherwise.

## P7 — package board, statuses, print batches, cards

- **DECISION-P7-1 (filing group = fulfillment method code):** the grill never named the filing dimension, and the physical workflow files paper by how it leaves the building. Nightly artifacts, reprints, and the channel dashboard all group by method code; per-order reprints file under `ORDER-#n`.
- **DECISION-P7-2 (nightly scope is stage-NEW):** the nightly batch prints packages nobody printed yet. Anything already PRINTED/PACKED reprints on demand per group or order — otherwise every night would regenerate the whole warehouse. Idempotency is a unique `runKey` per calendar day; a second trigger returns the existing batch (`replayed: true`).
- **DECISION-P7-3 (artifacts are payload snapshots, PDFs render on download):** `PrintArtifact.payload` freezes what was printed; the PDF is generated from that JSON at download time by a dependency-free writer (`lib/pdf.ts`, ladder rung 6 — no pdf package). Rendering reads only the snapshot, so printing can never touch `Package.stage` (G-002).
- **DECISION-P7-4 (split/regroup key discipline):** staff-split packages get `{key}#split-{rand}` grouping keys — the NEW-stage partial unique index holds, and finalize never merges new lines into a box staff deliberately carved out. Regroup moves lines to the oldest package and keeps the emptied sources as audited rows with `{key}#regrouped-{rand}` retired keys (history retained, never re-matched, hidden from the board).
- **DECISION-P7-5 (quantity splits clone the line; add-on lines move whole):** splitting part of a qty-N line decrements the source and creates a clone line with the same unit-price snapshot and option rows — order totals unchanged. Lines with add-ons only move whole, because per-line add-on units cannot be divided meaningfully.
- **DECISION-P7-6 (stage machine forward-only, channel-aware terminal):** optional stages mean skipping ahead is fine (NEW straight to SENT), moving backward never is; PICKUP channels finish at PICKED_UP, everything else at SENT. Per-package moves are version-guarded (409 to the loser); channel-wide bulk moves are one atomic conditional UPDATE audited with the count.

## P6 — admin ops hub, POS, customers, CSV import

- **DECISION-P6-1 (POS drafts live in OrderDraft, keyed per customer):** no new persistence layer. The POS stores its cart in the existing `OrderDraft` table under a synthetic guest key `pos|{customerId}`, so one in-progress POS cart exists per customer, fully isolated from that customer's own web draft. `OrderBuilder` gained `draftUrl` / `addressUrlFor` props; the storefront defaults are unchanged.
- **DECISION-P6-2 (POS checkout is server-trusted, not a CheckoutForm reuse):** the POS panel re-quotes on the server (`/api/admin/pos/quote`) and checkout re-derives everything via the same `createOrderFromCart` + `finalizeOrder` path as the storefront, with `expectedTotalCents` as a stale-screen guard. Cash/check post as POSTED payment rows attributed to the acting staff member (`pos.payment.post`, `pos.checkout` audits); no Stripe surface exists on the POS.
- **DECISION-P6-3 (bulk actions: per-id transactions, deterministic reports):** `/api/admin/orders/bulk` de-dupes and sorts ids, then runs each through the existing guarded finalize/discard state machine one transaction at a time. Racing bulk requests therefore split the set cleanly: every id lands in exactly one request's `done` and the other's `skipped` with a human reason. Bounded at 200 ids.
- **DECISION-P6-4 (CSV import re-stages at commit):** commit re-validates the raw CSV rather than trusting a client-echoed preview. Any invalid row refuses the whole file (409, nothing written); valid rows insert in ONE transaction; duplicates (in DB or earlier in the file) skip and are reported. Customer-import phone dedupe mirrors DECISION-P4-6: a phone owned elsewhere stores raw-only.
- **DECISION-P6-5 (numeric order search is exact):** in the order list, an all-digits query matches `orderNumber` exactly instead of fuzzy-matching references — searching "1" must return order #1, not every reference containing the digit. Customer search likewise only treats phone-shaped queries as phone lookups.
- **DECISION-P6-6 (dev Stripe page reads searchParams before the gateway):** `/dev/stripe-checkout` awaits `searchParams` before calling `getPaymentGateway()`, which marks the route dynamic so `next build`'s prerender never evaluates the mock-only gateway guard (which correctly throws under NODE_ENV=production — DECISION-P5-7).

## P8 — Shippo, rate margin, labels

- **DECISION-P8-1 (no Shippo account → mock wrapper, real path):** the harness has no Shippo token, and negotiated FedEx/UPS rates aren't reproducible in Shippo test mode anyway (plan risk #3). The wrapper is real REST when SHIPPO_API_TOKEN is set (fail-closed unless BOTH org carrier account ids are also set — R-183/184); without it, deterministic fixture rates (ZIP parity decides which carrier carries a surcharge, USPS only under 4.5kg/parcel) drive the same margin/label code paths, so S1 margin math is exact and repeatable.
- **DECISION-P8-2 (comparable services = each carrier's cheapest quote):** UR-003 says charge the highest and buy the cheaper, but never defined which service levels compare. Rule implemented: take every eligible carrier's CHEAPEST quoted service, charge the customer the highest of those, buy the overall cheapest, store charged/cost/margin plus the full comparison set on the Shipment row for P12 reconciliation.
- **DECISION-P8-3 (checkout shipping fails closed, placeholder retired):** a SHIPPING destination with no live rate is now a checkout error, not a guessed flat fee — money never falls back silently. The old "shipping.rates" setting no longer feeds checkout (kept only as a settings-tab relic).
- **DECISION-P8-4 (ShippingQuote may be unanchored):** checkout previews quote BEFORE any Order exists, so the P2 CHECK constraint (order or package required) was relaxed to forbid only both-at-once. Label purchases still anchor their quote to the package.

- **DECISION-P4-9 (guest sign-in drops the guest draft cookie):** login/register/logout clear `tomchei_guest_draft` so a shared-device guest draft (recipient names, addresses) can never re-attach to the next user. A guest's draft row is untouched; merging a guest cart into a customer account on sign-in is a product feature deferred with a note, not silently half-done.
